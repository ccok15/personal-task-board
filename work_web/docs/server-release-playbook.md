# 服务器发布必看手册

这份文档只服务于一件事：

- 后续每次把网站发布到阿里云 ECS 前，先看这里

目标也很明确：

- 保住线上数据库数据
- 不把本地测试数据带上去
- 不再让服务器因为本机编译而卡死或需要重启

## 2. 这套项目为什么不能再走“服务器本机编译”

之前已经验证过多次：

- 服务器本机执行 `next build`
- 或服务器本机 `docker build`

会把当前 ECS 的资源打满，典型表现是：

- SSH 卡死
- 网站失联
- 容器更新卡住
- 最后只能重启 ECS

所以这套项目的长期原则已经锁定：

- **服务器不负责编译**
- **服务器只负责接收已经构建好的镜像并替换 `app` 容器**

这是后续所有发布动作的底线。

## 4. 发布前必须确认的 7 件事

每次服务器发布前，必须先确认下面这些。

### 4.1 本地改动已经稳定

至少执行：

```bash
cd /Users/lsl/new_gpt/web_lsl/work_web
pnpm lint
pnpm build
```

如果这两步不过，不要发版。

### 4.2 小改动先做极简冒烟测试

看这里：

- `/Users/lsl/new_gpt/web_lsl/work_web/docs/simple-smoke-test.md:1`

默认只做极简测试，不要每次都全量回归。

### 4.3 确认本次是否改了数据库结构

如果只是前端样式、交互、文案、组件重排，通常：

- 不需要动数据库

如果改了 Prisma schema，就必须先判断：

- 是不是只新增字段
- 是不是有删除字段
- 是不是有重命名字段
- 会不会影响老数据

### 4.4 默认不允许直接清库

服务器发布时：

- 不允许 `prisma db push --accept-data-loss`
- 不允许删除线上表
- 不允许覆盖线上真实任务数据

### 4.5 默认不允许执行种子覆盖线上数据

服务器发布时：

- 不执行 `pnpm prisma:seed`
- 不允许把本地测试任务写到线上

线上真实任务数据只保留在线上数据库里，不从 GitHub 恢复。

### 4.6 确认 GitHub 只上传了代码，不上传数据库内容

正常情况下 GitHub 上传的是：

- 代码
- 样式
- Prisma schema
- 脚本
- 文档

不会上传：

- 线上任务数据
- 本地 PostgreSQL 内容

所以只要不把导出的数据库文件手动提交进仓库，GitHub 本身不会带走任务数据。

### 4.7 发布方式必须是“本地构建，远端替换 app”

这一条不能省略。

## 5. 数据库发布规则

这是后续最重要的一部分。

### 5.1 如果本次没有数据库变更

那就：

- 不碰数据库
- 只更新 `app` 容器

这是最安全的情况。

### 5.2 如果本次只是新增字段

优先原则：

- 用**最小改动**方式给线上数据库补字段
- 不删除任何老字段
- 不重建表
- 不清空数据

这次 `v1.2` 的 `Task.progress` 就是这个模式：

- 只新增 `progress integer`
- 已完成任务回填 `100`
- 原任务数据全部保留

这类变更可以接受。

### 5.3 如果本次涉及删除字段、重命名字段、改枚举

这类属于高风险发布。

默认规则：

- 先暂停
- 先写清楚迁移方案
- 先评估回滚方式

没有确认之前，不直接发线上。

### 5.4 已完成任务的补值原则

如果新增字段与任务状态强相关，可以接受这种“安全补值”：

- 例如 `DONE -> progress = 100`

前提是：

- 只做补值
- 不覆盖用户业务字段
- 不改变任务内容

### 5.5 数据库变更模板

以后每次涉及数据库结构调整，都先按这个模板写清楚：

1. **这次是否新增字段**
   - 字段名
   - 类型
   - 是否允许为空
   - 默认值是什么
2. **是否需要补历史数据**
   - 哪些旧记录要补
   - 补什么值
   - 补值条件是什么
3. **是否影响现有代码读取**
   - 前台会不会因为空值报错
   - 后台会不会因为旧数据缺字段报错
4. **发布后怎么验证**
   - 字段是否已经存在
   - 历史数据是否补齐
   - 页面是否能正常读取

推荐写成这种简短结构：

- 新增字段：`Task.progress`
- 类型：`integer`
- 默认值：`0`
- 历史补值：`DONE -> 100`
- 风险：低
- 验证 SQL：`information_schema.columns` + 关键行数检查

### 5.6 数据库高风险变更清单

下面这些变更默认都按高风险处理：

- 删除字段
- 重命名字段
- 改枚举值
- 改唯一约束
- 改外键关系
- 把可空改成非空
- 改默认值且会影响旧逻辑

遇到这些情况时，不要直接上线。

至少要补这三件事：

- 迁移方案
- 回滚方案
- 发布后验证方案

### 5.7 推荐验证 SQL

以后如果只是新增字段并补历史数据，优先准备这类验证：

```sql
select column_name, data_type
from information_schema.columns
where table_name = 'Task' and column_name = 'progress';
```

```sql
select count(*) as done_without_100
from "Task"
where status = 'DONE' and progress <> 100;
```

```sql
select id, title, status, progress
from "Task"
order by "updatedAt" desc
limit 20;
```

这样发布完能很快知道：

- 字段有没有加上
- 已完成任务有没有补齐
- 最近的数据有没有异常

### 第四步：执行远端应用发布

## 7. 这次已经验证过的经验

### 7.1 边压缩边通过 SSH 传镜像，不够稳

之前试过：

- `docker save | gzip | ssh ... docker load`

这条链路有过卡住的情况。

当前更稳的做法是：

- 本地先生成镜像压缩包
- 用 `scp` 传到服务器
- 再在服务器 `docker load`

如果以后发布再次卡住，优先切回这种文件式传输。

### 7.2 只更新 `app` 容器是正确的

当前站点已经证明：

- 数据库容器保持不动
- 只更新应用容器

可以把代码发布和数据保护拆开，这比整套 Compose 全重建安全得多。

### 7.3 线上数据库结构变更要走“最小补丁”

`progress` 字段这次能安全上线，原因是：

- 只是新增字段
- 不删字段
- 不清数据
- 只给 `DONE` 任务补 `100`

以后类似改动也应沿用这个原则。

### 7.4 本地测试数据不会因为 GitHub 自动进入线上

GitHub 保存的是仓库文件，不是本地 PostgreSQL 内容。

真正危险的是：

- 手动执行 `seed`
- 手动导入本地数据库
- 把导出数据文件提交到仓库

只要避开这三种情况，线上业务数据就是安全的。

## 8. 每次服务器发布必须回答的 5 个问题

每次发版前，先自己回答这 5 个问题：

1. 这次改动有没有改数据库结构？
2. 如果改了，是否只新增字段，是否会保留老数据？
3. 这次会不会运行 `seed`？
4. 这次是不是只更新 `app` 容器？
5. 这次发布完，我要怎么确认线上版本已经更新？

如果其中任意一个问题答不清楚，就不要直接发版。
